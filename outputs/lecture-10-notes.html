Only the bottom path observes the real system

The defect lives between components.

In the Electron export example, the renderer, preload script, and service layer each pass unit tests. The first real click exposes wrong path format, stuck progress, memory leakage, permission differences, and missing error propagation.

Unit-test view

Fast, precise, and isolated. Useful for local logic, but blind to the contracts between mocked components.

Every component looks correct on its own.
vs.

E2E view

Slower, but it exercises the real path through UI, bridge, service, operating system, and side effects.

Only end-to-end testing can prove absence of system-level defects.

Unit tests have systematic blind spots.

Isolation is the design philosophy that makes unit tests fast. It is also why they cannot catch defects created by wiring, shared state, real resources, or environment differences.

Interface mismatch

Renderer sends a relative file path while preload expects an absolute path.

State propagation

Schema changes but cached ORM entries still reflect the old table shape.

Resource lifecycle

File handles, database connections, and sockets leak across component boundaries.

Environment dependency

Mocked tests hide configuration, latency, permissions, and service availability.

E2E tests change how agents write code.

The lecture's overlooked point: when agents know the real flow will be run, they stop optimizing for the fastest green check and start respecting integration pressure.

Core concepts

Remember the gradient: each wider test layer catches a larger class of defects.

Component boundary defectsEach component passes alone, but the interaction between A and B fails.
Testing adequacy gradientUnit-detectable defects <= integration-detectable defects <= E2E-detectable defects.
Executable architectureArchitecture rules become lint checks or tests that fail automatically.
Review promotionRecurring review feedback becomes a permanent automated check.
Agent-oriented errorsFailure messages include what went wrong, why it matters, and exactly how to fix it.

Define boundaries before E2E.

Without architectural intent, a full-pipeline run only proves that the whole mess runs. For agent-built codebases, constraints need to exist on day one because agents copy existing patterns.

Rule: dependencies flow forward through fixed layers. Cross-domain concerns enter through explicit provider interfaces. Any other dependency is forbidden and mechanically enforced.

How to apply it

Require E2E for cross-component work.

Validation hierarchy: unit, integration, then E2E when multiple components change.

Automate architecture rules.

Convert constraints such as "renderer must not access filesystem" into lint or test checks.

Write repair instructions.

Failure messages need what, why, and fix steps so the agent can self-correct.

Grow the harness from reviews.

Every recurring defect category becomes a new permanent automated check.
