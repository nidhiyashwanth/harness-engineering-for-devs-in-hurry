Making the agent's runtime observable

Reliability is an evidence problem: the harness must expose what the system did and why the work should be accepted.

Without observability, every retry is a guess.

The agent can touch many files, run for 20 minutes, and still be unable to explain why tests fail or which critical paths changed. The harness, not the agent's memory, must preserve the evidence.

Looks correct

Code review shows what was written, but runtime traces show what actually ran.

Mystical evaluation

Without rubrics, acceptance depends on implicit assumptions and inconsistent judgment.

Blind retries

The agent fixes nearby code instead of the root cause because failure context is missing.

Handoff cliff

Missing state forces the next session to rediagnose, wasting 30-50% of session time.

Vague loop

Planner gives a fuzzy description. Generator guesses. Evaluator rejects with "it doesn't feel right." The cycle repeats 3-4 times.

About 45 minutes, barely acceptable output.

Observable loop

Sprint contract defines scope, standards, and exclusions. Runtime records style application. Rubric cites concrete evidence.

About 15 minutes, high-quality output: 3x efficiency.

Observability has two layers.

Adding logs is not enough. Runtime signals explain behavior; process artifacts explain intent and acceptance.

Runtime observability

What did the system do?

Logs, traces, process states, health checks, data flow, resources, errors, and side effects.

Process observability

Why should this pass?

Plans, sprint contracts, scoring rubrics, exclusions, acceptance criteria, and evaluator decisions.

CONTRACT + SIGNALS -> REVIEW -> SPECIFIC VERDICT -> NEXT FIX

Core concepts

A good harness records enough decision context to replay what happened and why.

Task traceComplete decision-path record from task start to completion, like request tracing for agent work.
Sprint contractShort-term agreement defining scope, verification standards, and exclusions before coding starts.
Evaluator rubricStructured scoring that turns "is it good?" into evidence-based dimensions.
Runtime signalsObjective system behavior: logs, traces, health, data flow, resources, exceptions.
Layered observabilityRuntime and process evidence designed together so each reinforces the other.

The agent cannot solve this by printing logs.

Unknown unknowns

Agents only log what they think matters. That is usually not enough.

Inconsistent formats

Different sessions produce different logs, making systematic analysis brittle.

Process gaps

Sprint contracts and rubrics are structured harness artifacts, not print statements.

How to build it

Collect runtime signals automatically.

Capture lifecycle, critical paths, data flow, resources, exceptions, and full error context.

Negotiate sprint contracts.

Define scope, verification standards, and exclusions before the generator writes code.

Use evaluator rubrics.

Score dimensions such as correctness, architecture, test coverage, visual design, and code quality.

Standardize traces.

Use OpenTelemetry: session trace, task span, verification sub-spans, and standard attributes.

# Sprint Contract: Dark Mode Support
Scope:
- Modify theme toggle component
- Update global CSS variables
- Add dark mode tests

Verification:
- Visual regression tests pass
- Main E2E flow passes
- No flash of unstyled content

Exclusions:
- No print styles
- No third-party dark mode

Rubrics make evaluation reproducible.

The evaluator should cite specific evidence, not vibes. For example: "Button color contrast is insufficient: WCAG AA requires 4.5:1, measured 2.1:1."
